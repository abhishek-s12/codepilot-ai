Codexa design system

The visual language behind the landing site, workspace, and dashboard — one token set, so a call-graph node on the marketing page and a call-graph node in the editor read as the same product.

Principles

01

Four rules that shaped every screen. Break them deliberately, not by default.

A
Structure is information
File-path breadcrumbs (codexa/index.tsx) replace numbered steps everywhere, because Codexa's whole subject is a file tree — the device should come from the product, not a generic template.
B
Color carries meaning, not decoration
Amber marks execution / trace / agent activity. Cyan marks data, embeddings, and search. Green and red are reserved only for real diffs — never used decoratively elsewhere, so when they appear, they mean something.
C
Real content over placeholder content
Every code sample, trace, and citation in this system uses realistic file names and line numbers from an actual auth flow — not lorem or foo/bar.
D
Quiet by default, loud on purpose
The interface stays near-black and low-contrast at rest. Amber is spent deliberately — on the one active trace, the one modified file, the one thing that needs attention.

Motion

Voice & writing

08

Interface copy rules, condensed.

1
Name what people control
"Connect repository," not "configure git provider." A person clones a repo — they don't configure a VCS integration.
2
Keep the verb through the whole flow
A button that says "Accept patch" is followed by a state that says "Patch accepted" — never "Applied" or "Done."
3
Errors state fact, not feeling
"Index stale — 6 days ago," not "Oops, something's out of date!" The interface reports; it doesn't apologize.
