SECTION 1 — PRE-DEPLOYMENT CHECKLIST
==============================================================

Before you push anything to production, verify each of these:

  [ ] backend/.env is NOT committed to git (check .gitignore)
  [ ] backend/.env.example is committed (no real secrets — just keys)
  [ ] CORS origins in settings.py includes your Vercel domain
  [ ] Frontend uses VITE_API_URL env var (not hardcoded localhost)
  [ ] ChromaDB chroma_db/ folder is in .gitignore
  [ ] repos/ folder is in .gitignore (cloned repos must not be committed)
  [ ] venv/ folder is in .gitignore
  [ ] node_modules/ is in .gitignore
  [ ] dist/ folder is in .gitignore (Vercel builds this itself)
  [ ] All API keys are stored in environment variables, not in code
  [ ] requirements.txt is up to date (pip freeze > requirements.txt)
  [ ] npm run build succeeds locally with no errors
  [ ] python -c "import main" succeeds from the backend/ directory

SECTION 2 — TOOLING SETUP (BEFORE DEPLOYING)
==============================================================

--------------------------------------------------------------
2A. RUFF — Python Linting + Formatting (replaces Black + Flake8)
--------------------------------------------------------------

Install:
  pip install ruff

Add a ruff.toml to the backend/ folder:

  [tool.ruff]
  line-length = 88
  target-version = "py311"

  [tool.ruff.lint]
  select = ["E", "F", "I"]   # pycodestyle, pyflakes, isort

  [tool.ruff.format]
  quote-style = "double"

Run locally:
  ruff check .          # check for lint errors
  ruff format .         # auto-format all Python files
  ruff format --check . # check formatting without making changes (for CI)

Add to backend CI (.github/workflows/backend.yml):
  - name: Lint (Ruff)
    run: |
      pip install ruff
      ruff check .
      ruff format --check .

DO NOT add Black separately — Ruff's formatter is Black-compatible.

--------------------------------------------------------------
2B. PRETTIER — JavaScript/JSX Formatting
--------------------------------------------------------------

Install (from inside the frontend/ folder):
  npm install --save-dev prettier eslint-config-prettier

Create frontend/.prettierrc:
  {
    "semi": false,
    "singleQuote": true,
    "tabWidth": 2,
    "trailingComma": "es5",
    "printWidth": 80,
    "plugins": []
  }

Create frontend/.prettierignore:
  dist
  node_modules
  public

Run locally:
  npx prettier --write .        # format all files
  npx prettier --check .        # check without changing (for CI)

Add to eslint.config.js to avoid ESLint/Prettier conflicts:
  import prettierConfig from 'eslint-config-prettier'
  // add prettierConfig to the extends array

Add to frontend CI (.github/workflows/frontend.yml):
  - name: Format check (Prettier)
    run: npx prettier --check .

--------------------------------------------------------------
2C. ESLINT — Already Configured
--------------------------------------------------------------

Your eslint.config.js already includes:
  - @eslint/js recommended rules
  - eslint-plugin-react-hooks (prevents hook rule violations)
  - eslint-plugin-react-refresh (HMR safety)

No changes needed. Just make sure the CI runs it:
  - name: Lint (ESLint)
    run: npm run lint

This is already defined in your package.json as "lint": "eslint ."

SECTION 3 — DOCKERIZING THE BACKEND (FOR RENDER)
==============================================================

Create backend/Dockerfile:

  FROM python:3.11-slim

  WORKDIR /app

  # System deps for sentence-transformers and ChromaDB
  RUN apt-get update && apt-get install -y \
      build-essential \
      git \
      && rm -rf /var/lib/apt/lists/*

  COPY requirements.txt .
  RUN pip install --no-cache-dir -r requirements.txt

  COPY . .

  # Create directories that the app writes to at runtime
  RUN mkdir -p repos chroma_db

  EXPOSE 8000

  CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Create backend/.dockerignore:
  venv/
  __pycache__/
  .env
  repos/
  chroma_db/
  *.pyc
  .git

Build and test locally:
  docker build -t codepilot-backend .
  docker run -p 8000:8000 --env-file .env codepilot-backend

Verify at: http://localhost:8000/
Expected response: {"message": "CodePilot AI Running"}

NOTE: ChromaDB stores data in chroma_db/ which is ephemeral in Docker.
On Render, this resets on every redeploy. See Section 8 (Don'ts) for
how to handle persistent storage.

SECTION 4 — DEPLOYING FRONTEND ON VERCEL
==============================================================

Step 1 — Update frontend to use environment variable for API URL
  In frontend/src/services/api.js, replace any hardcoded URLs:
    BEFORE: const BASE_URL = "http://localhost:8000"
    AFTER:  const BASE_URL = import.meta.env.VITE_API_URL || "http://localhost:8000"

Step 2 — Connect to Vercel
  a. Go to https://vercel.com and sign in with GitHub
  b. Click "Add New Project"
  c. Import your GitHub repo: abhishek-s12/codepilot-ai
  d. Vercel will auto-detect it as a Vite project

Step 3 — Configure Vercel project settings
  Framework Preset : Vite
  Root Directory   : frontend          ← IMPORTANT: set this
  Build Command    : npm run build
  Output Directory : dist
  Install Command  : npm install

Step 4 — Set environment variables in Vercel dashboard
  Key             : VITE_API_URL
  Value           : https://your-render-backend.onrender.com
  (Set after your Render backend is deployed)

Step 5 — Deploy
  Click Deploy. Vercel builds and serves your frontend automatically.
  Every push to main will trigger a new deploy.
  Every pull request gets a preview URL automatically.

Your frontend URL will look like:
  https://codepilot-ai.vercel.app


==============================================================
SECTION 5 — DEPLOYING BACKEND ON RENDER
==============================================================

Step 1 — Push your backend/Dockerfile to GitHub (Section 3 must be done)

Step 2 — Create a new Web Service on Render
  a. Go to https://render.com and sign in
  b. Click "New" → "Web Service"
  c. Connect your GitHub repo: abhishek-s12/codepilot-ai

Step 3 — Configure the Render service
  Name            : codepilot-backend
  Root Directory  : backend             ← IMPORTANT: set this
  Environment     : Docker
  Dockerfile Path : ./Dockerfile        (relative to Root Directory)
  Region          : Oregon (US West) or closest to you
  Instance Type   : Free (or Starter for always-on)

Step 4 — Set environment variables in Render dashboard
  LLM_API_KEY       : your-openrouter-api-key
  LLM_MODEL         : openai/gpt-4o-mini   (or preferred model)
  LLM_BASE_URL      : https://openrouter.ai/api/v1
  LLM_APP_NAME      : CodePilot AI
  LLM_SITE_URL      : https://codepilot-ai.vercel.app
  CORS_ORIGINS      : ["https://codepilot-ai.vercel.app"]

  NOTE: CORS_ORIGINS must be updated in settings.py to read from env:
    cors_origins: list[str] = Field(
        default=["http://localhost:5173"],
        validation_alias=AliasChoices("CORS_ORIGINS"),
    )

Step 5 — Deploy
  Click "Create Web Service". Render builds the Docker image and deploys.
  Your backend URL will look like:
    https://codepilot-backend.onrender.com

Step 6 — Update Vercel with the backend URL
  Go back to Vercel → Environment Variables
  Update VITE_API_URL to: https://codepilot-backend.onrender.com
  Redeploy the frontend.

Step 7 — Verify the connection
  Open your Vercel URL in the browser.
  Clone a repo and try indexing it.
  Watch the Render logs for any errors.

SECTION 7 — CI/CD PIPELINE NOTES
==============================================================

Current GitHub Actions workflows:
  .github/workflows/backend.yml   — installs deps, verifies import
  .github/workflows/frontend.yml  — installs deps, runs npm build

Recommended additions to backend.yml:
  - Ruff lint check (ruff check .)
  - Ruff format check (ruff format --check .)
  - Add pytest step once you have real test files (not just ad-hoc scripts)

Recommended additions to frontend.yml:
  - ESLint step (npm run lint)
  - Prettier check (npx prettier --check .)
  - Add pull_request trigger (currently only triggers on push to main)

Render auto-deploy:
  Render watches your GitHub repo and auto-deploys when main is updated.
  You can disable this and use manual deploys in the Render dashboard.

Vercel auto-deploy:
  Vercel auto-deploys on every push to main.
  PRs get automatic preview URLs — very useful for reviewing changes.

SECTION 8 — DOS AND DON'TS
==============================================================

--------------------------------------------------------------
DOs
--------------------------------------------------------------

DO   use VITE_API_URL env var in frontend instead of hardcoding localhost
DO   add your Vercel domain to CORS_ORIGINS before deploying
DO   keep .env in .gitignore — always use .env.example for documentation
DO   use ruff for Python linting AND formatting (one tool, not two)
DO   run "npm run build" locally before deploying to catch build errors
DO   set the Root Directory in both Vercel and Render (frontend/ and backend/)
DO   use Render's "Suspend" feature on the free tier to avoid idle charges
DO   keep requirements.txt updated after adding new packages
DO   test the Docker build locally before pushing to Render
DO   use Vercel Preview Deployments to test changes before merging to main
DO   add a health check endpoint (/health) to your FastAPI for Render monitoring
DO   use @lru_cache on get_settings() to avoid re-reading .env on every request
     (this is already done — keep it)

--------------------------------------------------------------
DON'Ts
--------------------------------------------------------------

DON'T commit .env files — they contain your API keys
DON'T commit chroma_db/ or repos/ — these are runtime data, not source code
DON'T commit venv/ or node_modules/ — these are always regenerated
DON'T hardcode "localhost:8000" anywhere in frontend production code
DON'T use the Render free tier for a production app that needs to stay
      awake — free tier spins down after 15 minutes of inactivity
      (first request after spin-down takes ~30 seconds)
DON'T rely on ChromaDB's local disk persistence on Render — it resets on
      every redeploy. For production, consider a persistent volume or
      migrate to a hosted vector DB (see Version 2 suggestions)
DON'T use "allow_origins=['*']" in production CORS config — always specify
      exact origins for security
DON'T run both Black and Ruff — they conflict on formatting opinions
DON'T put large repos in the repos/ folder and commit them — always
      .gitignore the repos/ directory
DON'T skip the "Root Directory" setting in Vercel/Render — without it,
      the build will fail because it won't find the right package.json
      or Dockerfile
DON'T use the main branch directly for experimental features — use
      feature branches and PRs so preview deployments protect main

SECTION 9 — VERSION 2 FEATURE SUGGESTIONS
==============================================================

These are features that would meaningfully upgrade CodePilot AI from
a developer tool into a production-grade code intelligence platform.

--------------------------------------------------------------
CORE INTELLIGENCE UPGRADES
--------------------------------------------------------------

1. PERSISTENT VECTOR DATABASE
   Problem: ChromaDB on Render resets on every redeploy. Users lose their
            indexed repositories and have to re-index on every deployment.
   Suggestion: Migrate to a hosted vector database.
   Options:
     - Pinecone (managed, free tier available)
     - Qdrant Cloud (open-source, self-hostable)
     - Weaviate Cloud (GraphQL API, good for structured queries)
   Impact: HIGH — solves the biggest production reliability issue

2. MULTI-FILE DIFF REVIEW
   Problem: The current /review endpoint likely reviews one file at a time.
   Suggestion: Accept a git diff (unified diff format) and review all changed
               files together with cross-file impact analysis.
   Impact: HIGH — makes CodePilot genuinely useful in PR workflows

3. STREAMING AI RESPONSES
   Problem: AI responses currently block until the full response is ready,
            causing a poor UX on long explanations.
   Suggestion: Use FastAPI StreamingResponse + SSE (Server-Sent Events)
               on the backend and ReadableStream on the frontend to stream
               tokens as they arrive.
   Impact: HIGH — dramatically improves perceived performance

4. MULTI-REPO SUPPORT
   Problem: Only one repository can be indexed and queried at a time.
   Suggestion: Let users maintain a list of indexed repos with namespaced
               ChromaDB collections (one collection per repo). Add a repo
               switcher in the sidebar.
   Impact: MEDIUM — important for developers working across multiple projects

--------------------------------------------------------------
DEVELOPER EXPERIENCE UPGRADES
--------------------------------------------------------------

5. VS CODE EXTENSION
   Suggestion: Package the "Ask AI" and "Explain Flow" features as a VS Code
               extension that calls the same FastAPI backend. Users could
               right-click any function and ask questions about it inline.
   Impact: HIGH — dramatically increases adoption and daily usage

6. GITHUB PR INTEGRATION (WEBHOOK)
   Suggestion: Add a GitHub webhook listener. When a PR is opened, CodePilot
               automatically runs architecture review + diff review and posts
               a summary comment on the PR.
   Implementation: FastAPI webhook endpoint + GitHub API for posting comments
   Impact: HIGH — turns CodePilot into a team tool, not just a personal tool

7. AUTHENTICATION + USER ACCOUNTS
   Suggestion: Add auth so multiple users can have their own indexed repos.
   Options:
     - Clerk (easiest, React SDK available)
     - Supabase Auth (free tier, works well with FastAPI)
     - Auth0
   Impact: MEDIUM — required before sharing with a team

8. REPO INDEXING PROGRESS INDICATOR
   Problem: Indexing a large repo takes time with no feedback on progress.
   Suggestion: Use WebSocket or SSE to stream indexing progress from backend
               to frontend. Show a progress bar with current file being indexed.
   Impact: MEDIUM — improves UX significantly for larger repos

--------------------------------------------------------------
ANALYSIS FEATURE UPGRADES
--------------------------------------------------------------

9. DEPENDENCY GRAPH
   Suggestion: Beyond the import graph, show a visual D3.js or Cytoscape.js
               graph of module dependencies. Let users click nodes to navigate
               to that module's code.
   Impact: MEDIUM — visual tools are highly memorable and shareable

10. DEAD CODE DETECTION
    Suggestion: Cross-reference all defined functions against all call sites.
                Functions with zero callers are flagged as potentially dead code.
                Show them in a dedicated "Dead Code" panel.
    Implementation: Already have call graph data — just need filtering logic.
    Impact: MEDIUM — easy to implement given existing call graph infrastructure

11. COMPLEXITY METRICS
    Suggestion: Run cyclomatic complexity analysis on each function using
                the AST. Flag functions above a threshold (e.g., complexity > 10)
                in the UI with a warning badge.
    Libraries: radon (Python) for cyclomatic complexity
    Impact: MEDIUM — adds quantitative quality metrics

12. TEST COVERAGE HINTS
    Suggestion: After indexing, identify functions that have no corresponding
                test file/function. Generate suggested test stubs using the LLM.
    Impact: MEDIUM — very useful for projects with poor test coverage

--------------------------------------------------------------
DEPLOYMENT AND INFRASTRUCTURE UPGRADES
--------------------------------------------------------------

13. DOCKER COMPOSE FOR LOCAL DEV
    Suggestion: Add a docker-compose.yml at the root that starts both frontend
                (on port 5173) and backend (on port 8000) together with one command:
                  docker-compose up
    Impact: LOW-MEDIUM — improves onboarding for new contributors

14. RATE LIMITING
    Suggestion: Add rate limiting to the AI endpoints (/ai/ask) to prevent
                runaway API costs if the app is public.
    Library: slowapi (FastAPI-compatible rate limiter)
    Impact: MEDIUM — important before making the app publicly accessible

15. CACHING LAYER
    Suggestion: Cache LLM responses for identical questions about the same repo.
                If user asks "explain the flow of clone_repo" twice, serve the
                cached answer instead of calling the LLM again.
    Options: Redis (Render has a Redis addon), or simple in-memory dict with TTL
    Impact: MEDIUM — reduces API costs significantly

--------------------------------------------------------------
PRIORITY ORDER FOR VERSION 2
--------------------------------------------------------------

  P1 (Do First):
    - Persistent vector DB (Pinecone/Qdrant) — fixes core reliability
    - Streaming AI responses — biggest UX improvement
    - GitHub PR integration — turns it into a team tool

  P2 (Do Second):
    - Authentication + user accounts
    - Multi-repo support
    - VS Code extension

  P3 (Do Later):
    - Dependency graph visualization
    - Dead code detection
    - Complexity metrics
    - Rate limiting + caching
