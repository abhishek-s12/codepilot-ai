PROBLEM 1: ChromaDB — Embedding Function Conflict at Startup
==============================================================

SYMPTOM:
  When the backend started, ChromaDB threw a conflict/ValueError
  because the existing persisted collection was created without
  an embedding function, but the new code tried to attach a
  LocalEmbeddingFunction (using sentence-transformers).

ROOT CAUSE:
  The original chroma_service.py used a top-level global variable:
    collection = client.get_or_create_collection(name="code_chunks")
  This was initialized once at import time with no embedding function.
  Switching to LocalEmbeddingFunction caused a mismatch with the
  already-persisted collection on disk.

FIX APPLIED (backend/vector_store/chroma_service.py):
  - Replaced the global `collection` variable with a `get_collection()`
    factory function that always creates or fetches the collection with
    LocalEmbeddingFunction attached.
  - Added error handling: if a ValueError about "embedding function conflict"
    occurs, the old collection is deleted and recreated cleanly.
  - The LocalEmbeddingFunction class wraps sentence-transformers:
      class LocalEmbeddingFunction(chromadb.EmbeddingFunction):
          def __call__(self, input):
              return [generate_embedding(doc) for doc in input]
  - Every function (add_chunk, search_chunks, reset_collection) now
    calls get_collection() instead of referencing a stale global.

PROBLEM 2: Call Graph — No Cross-File Function Resolution
==============================================================

SYMPTOM:
  The repository call graph only showed function names as plain strings
  (e.g., "clone_repository") with no file context. When two files defined
  functions with the same name, it was impossible to tell them apart.
  Callee functions from other modules were silently dropped.

ROOT CAUSE:
  The original build_repository_call_graph() in
  repository_call_graph_service.py worked like this:
    - Collect all function names from all .py files into one flat dict
    - Filter callees to only keep those that exist in the same flat dict
  This meant:
    a) Cross-file calls were lost unless the name exactly matched another
       top-level function in any file (no module context).
    b) Qualified names like "ClassName.method" were never tracked.

FIX APPLIED (backend/services/repository_call_graph_service.py):
  Completely rewrote the resolution logic with 4-stage resolution:

  Stage 1 — Generate import graph
    Call generate_repository_graph(repo_path) first to know which
    modules each file imports.

  Stage 2 — Extract raw calls per file
    For each .py file, extract function calls using extract_function_calls()
    and track them as (file_rel_path, func_name) -> [raw_call_names].
    Also build:
      - defined_functions_by_file: {file -> set of defined func names}
      - all_defined_functions: {func_name -> list of files that define it}

  Stage 3 — Resolve each call to a fully-qualified name:
    Case A: Call defined in same file -> file::func
    Case B: call is ClassName.method and class is in same file -> file::call
    Case C: Match call against imported modules -> other_file::func
    Case D: If the function is uniquely defined in one file -> that_file::func

  Result: All graph keys and values are now in "file.py::function" format.
  Example output:
    backend/api/repository.py::clone_repo -> [backend/services/repo_service.py::clone_repository]

PROBLEM 3: Call Graph — Nested Functions Incorrectly Attributed
==============================================================

SYMPTOM:
  Functions defined inside other functions were being added to the graph
  as top-level functions. Calls inside inner functions were attributed
  to the outer function, creating incorrect call relationships.

ROOT CAUSE:
  The original extract_function_calls() used ast.walk(tree) for the
  outer loop AND ast.walk(node) inside each FunctionDef. ast.walk()
  descends into everything including nested FunctionDefs and ClassDefs.

FIX APPLIED (backend/services/call_graph_service.py):
  - Added walk_exclude_nested_scopes(node) — a custom AST walker that
    yields nodes from a function body but stops descending into any
    child FunctionDef, AsyncFunctionDef, or ClassDef it finds.
  - Replaced ast.walk(node) with walk_exclude_nested_scopes(node)
    inside the call extraction loop.
  - Rewrote the outer visitor as a proper CallGraphVisitor(ast.NodeVisitor)
    with a class_stack to track class context:
      - visit_ClassDef: pushes/pops class name onto stack
      - visit_FunctionDef: names functions as "ClassName.method" if inside
        a class, otherwise just "function_name"
      - visit_AsyncFunctionDef: delegates to visit_FunctionDef

PROBLEM 4: Repository Graph Service — Wrong Path Keys
==============================================================

SYMPTOM:
  The import graph built by generate_repository_graph() was using the
  bare filename (e.g., "repository.py") as dict keys. The call graph
  resolver expected relative paths (e.g., "backend/api/repository.py").
  This mismatch meant cross-file resolution in Stage 3 always failed.

ROOT CAUSE:
  The original code stored the import graph as:
    graph[file] = imports   # file was just the filename, not a path

FIX APPLIED (backend/services/repository_graph_service.py):
  Changed the key to use a relative path:
    relative_path = os.path.relpath(file_path, repo_path).replace("\\", "/")
    graph[relative_path] = imports
  Now both the import graph and the call graph use the same key format,
  enabling accurate cross-file call resolution.

PROBLEM 7: openai-cookbook Clone Fails on Windows (NTFS)
==============================================================

SYMPTOM:
  Clicking "Fill Sample" in the UI (which used openai-cookbook as
  the sample URL) resulted in an error:
    git.exc.GitCommandError: error: invalid path
    'examples/data/hotel_invoices/extracted_invoice_json /20190119_002_extracted.json'
    fatal: unable to checkout working tree

ROOT CAUSE:
  The openai-cookbook repository contains a file with a trailing space
  in a directory name ("extracted_invoice_json /" with a space before
  the slash). Windows NTFS does not allow filenames/directories with
  trailing spaces, so git checkout fails.

FIX APPLIED:
  - Changed the sample repository URL in App.jsx from:
      "https://github.com/openai/openai-cookbook"
    to the project's own repository:
      "https://github.com/abhishek-s12/codepilot-ai"
  - This avoids the NTFS limitation entirely.
  - Note: Any repository with Windows-incompatible filenames will
    fail the same way. This is a git/Windows limitation, not a code bug.

PROBLEM 8: Missing settings.py — No Centralized Configuration
==============================================================

SYMPTOM:
  The backend had no central configuration system. API keys and
  URLs were hardcoded or read inconsistently using os.environ.
  Starting the server with a fresh .env produced silent failures.

ROOT CAUSE:
  No settings module existed. Each service that needed configuration
  did its own ad-hoc environment variable reading.

FIX APPLIED (backend/settings.py — new file):
  Created a Pydantic BaseSettings class that:
    - Reads from backend/.env automatically
    - Supports both new and legacy env var names via AliasChoices:
        LLM_API_KEY or OPENROUTER_API_KEY (both work)
        LLM_BASE_URL or OPENROUTER_BASE_URL
        LLM_MODEL or OPENROUTER_MODEL
    - Has sensible defaults (OpenRouter base URL, gpt-4o-mini model)
    - Is cached via @lru_cache so .env is only read once
  All services now import: from settings import get_settings

PROBLEM 9: Missing api/schemas.py — No Request Validation Models
==============================================================

SYMPTOM:
  API endpoints were defining inline Pydantic models or using
  raw dict parameters, making the code messy and error-prone.
  Some endpoints accepted invalid data without proper validation.

ROOT CAUSE:
  No shared schemas file existed. Each api/*.py file either
  defined its own models or lacked proper type checking.

FIX APPLIED (backend/api/schemas.py — new file):
  Created three shared Pydantic models:
    class RepositoryCloneRequest(BaseModel):
        repo_url: HttpUrl       # validates it's a proper URL
    class RepositoryPathRequest(BaseModel):
        repo_path: str          # for indexer, architecture, call-graph, flow
    class QuestionRequest(BaseModel):
        question: str           # for AI ask endpoint
  All API routers now import from api.schemas.

  interactive elements, no state feedback, no source attribution.
